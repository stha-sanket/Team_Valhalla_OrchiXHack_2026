\section{Entity Overview}
MongoDB holds ten collections. Seven carry the product domain:

\begin{itemize}
  \item \textbf{User} --- account identity, role, avatar, and a permanent
        embedded log of earned milestones.
  \item \textbf{VisitingPlace} --- a site with central coordinates, badge
        image, a per-place geofence radius stored as
        \code{visit\_threshold\_meters}, and a denormalised
        \code{visitor\_count}.
  \item \textbf{VisitingRoutes} --- the ordered waypoints of a place; each
        has GPS coordinates, media, and a type
        (\code{start}, \code{node}, \code{milestone}, \code{side\_quest},
        \code{end}) that determines behaviour on arrival.
  \item \textbf{UserProgress} --- one record per (user, place) holding a
        visited flag per waypoint; reset-able for revisits.
  \item \textbf{ArPoints} --- one append-only points ledger per user.
  \item \textbf{Quiz} / \textbf{QuizAttempt} --- one five-question quiz per
        place and one permanent attempt per (user, quiz).
\end{itemize}

Three support authentication and access control: \textbf{Admin} (a separate
collection from User), \textbf{Session}, and \textbf{Role}. Foreign keys are
stored as plain string IDs rather than ObjectId references --- a deliberate
choice that keeps the service layer free of ODM-specific types.

\section{Key Design Decisions}

\subsection{Separate User and Admin collections}
Administrators live in their own collection, so
\code{requireAuth('admin')} queries different data entirely; a
role-checking bug in application logic cannot escalate a regular user,
because the privilege boundary exists at the data layer.

\subsection{An append-only points ledger}
\code{ArPoints} is a transaction log, not a mutable balance. Every award
appends an entry keyed by \code{(source, ref\_id)}; \code{awardPoints()}
refuses a duplicate key, so re-confirming a checkpoint after a GPS glitch
or a progress reset can never farm points. Redemptions append negative
entries with unique references (they may legitimately repeat). A running
\code{total} is kept in sync for O(1) balance reads.

\subsection{Permanent achievements over reset-able progress}
\code{UserProgress} is deliberately disposable --- revisiting a place resets
it. Achievements therefore live elsewhere: milestones are embedded on the
User document at confirmation time, and place completion is recorded as a
\code{place\_complete} ledger entry. Badge computation checks the permanent
records as well as live progress, so a badge earned once survives any
number of resets.

\subsection{Denormalised visitor counting}
Showing ``visited by $N$ explorers'' by scanning progress records would
cost O(users) per read. Instead, \code{visitor\_count} on the place is
incremented exactly once per user, at first completion --- the idempotent
place-completion award doubles as the uniqueness signal --- making the read
O(1) at any scale.
